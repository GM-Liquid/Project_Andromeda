# Технический дизайн: контракт данных Project Andromeda 0.5

**Дата:** 2026-07-22  
**Статус:** рекомендован к утверждению; решения из раздела 11 требуют подтверждения владельца  
**Версия системы:** `0.4.04.4 → 0.4.05.0` (разряд RULES)

## 1. Контекст и цель

Канонические правила 0.5 вводят Накал, временные ячейки стресса от архетипа,
Истощение и четыре ранговые версии сигнатурной способности. Одновременно обычные оружие,
броня и снаряжение перестают быть самостоятельными механическими сущностями.

Цель этого документа — зафиксировать единый контракт данных и публичных методов для
последующей реализации в Foundry. Документ не меняет канон, runtime, каталоги или публичные
маршруты.

Канонические инварианты:

- Накал начинается с `0`, не имеет максимума и сгорает в конце боя;
- временные ячейки архетипа заполняются раньше основных, не восстанавливаются внутри сцены
  и исчезают в её конце;
- Истощение `X` складывается и снижает основной максимум стресса до `ранг × 5 − X`;
- только сигнатурная способность имеет четыре версии; покупные способности остаются
  плоскими записями;
- существующие пользовательские документы нельзя удалять или разрушительно пересоздавать
  без отдельного решения.

## 2. Накал: новое поле, а не переименование Flux

Накал хранится в новом `system.heat`. Поле `system.flux` не переименовывается и его значение
не переносится: в текущем runtime оно не участвует в механике и не имеет гарантированно той
же семантики.

```json
{
  "heat": {
    "value": 0,
    "gloryConversionUsed": false
  }
}
```

Правила совместимости:

- миграция создаёт `system.heat` со значением `0`;
- `system.flux` остаётся инертным legacy-полем минимум на переходный релиз;
- миграция не удаляет и не обнуляет `system.flux`;
- вторичная полоса токена переезжает с `system.flux.value` на `system.heat.value`;
- `heat.value` всегда является целым числом `>= 0`; максимума нет;
- `gloryConversionUsed` сбрасывается вместе с Накалом в начале и конце боя.

## 3. Контракт стресса

Сохраняемые поля Actor:

```text
system.stress.value                       заполненный основной стресс
system.stress.marked                      существующая разметка ячеек
system.stress.exhaustion                  суммарное Истощение X
system.stress.temporary.value             заполненные временные ячейки текущей сцены
```

Производные поля, которые не сохраняются:

```text
system.stress.baseMax
system.stress.max
system.stress.temporary.max
system.stress.total.value
system.stress.total.max
```

### 3.1. Временные ячейки архетипа

`system.stress.temporary.value` хранит число **заполненных**, а не оставшихся временных
ячеек. Их максимум вычисляется из единственного предмета-архетипа:

```text
temporary.max = currentRank × archetype.system.stressBonusPerRank
```

Если действующего архетипа нет, максимум равен `0`. `stressBonusPerRank` хранится на Item
архетипа; для старой записи без поля допустим совместимый fallback `2`, но каноническая
синхронизация должна материализовать настоящее значение.

При получении урона сначала растёт `temporary.value`, затем `stress.value`. Обычное
восстановление очищает только основной стресс. В начале сцены основной и временный
заполненный стресс сбрасываются в `0`; в конце сцены временный заполненный стресс также
сбрасывается в `0`.

Старые `system.temphealth` и `system.forceShield` для новой модели не переиспользуются:
их значения остаются legacy, но не участвуют в расчётах 0.5.

### 3.2. Истощение

`system.stress.exhaustion` — целое число `>= 0`. Хранится полный суммарный `X`, даже если он
превышает базовый максимум: это позволяет позднее корректно снять только часть накопленного
Истощения.

```text
baseMax = getBaseStressByRank(actor.type, currentRank)
stress.max = max(0, baseMax − exhaustion)
temporary.max = currentRank × stressBonusPerRank
total.value = temporary.value + stress.value
total.max = temporary.max + stress.max
```

Истощение уменьшает только основной максимум. При увеличении `X` заполненный основной
стресс ограничивается новым максимумом; если после пересчёта достигнут полный стресс,
применяется обычное выбытие. Уменьшение `X` увеличивает максимум, но не восстанавливает и
не наносит стресс.

## 4. Существующие weapon, armor и equipment

Базовая миграция не преобразует и не удаляет пользовательские `weapon`, `armor` и
`equipment`. Они сохраняются как legacy-документы, но вся их автоматическая механика 0.4
отключается.

Миграция добавляет детерминированный флаг:

```json
{
  "flags": {
    "project-andromeda": {
      "legacy05": {
        "sourceRuleset": "0.4",
        "originalType": "weapon",
        "mechanicsDisabled": true
      }
    }
  }
}
```

Для legacy-предметов сохраняются UUID, папка, имя, описание, количество, equipped,
пользовательские правки и библиотечные ссылки. Они:

- показываются в отдельной группе `Legacy 0.4`;
- не дают стресс, защиты, скорость и другие числовые бонусы;
- не создают канонические атаки и не применяют rules-эффекты;
- исключаются из автосинхронизации и orphan-cleanup;
- остаются зарегистрированными типами для загрузки старых миров, но не предлагаются при
  создании новых предметов 0.5.

Автоматическое преобразование в способности или артефакты отклонено: для homebrew нет
надёжного соответствия, а тип Foundry Item неизменяем и потребовал бы разрушительного
пересоздания документа. Возможный конвертер должен быть отдельным opt-in инструментом с
предпросмотром.

## 5. Сигнатурная способность

Сигнатурка представлена одним actor-owned Item существующего типа
`trait-source-ability`. Item сохраняет стабильный UUID, ссылку на компендиум и флаг
`grantedByArchetype`. Четыре версии не создаются как четыре отдельных Item.

Общие поля хранятся на верхнем уровне, меняющиеся по рангу — в `system.versions`:

```json
{
  "skill": "blizhniy_boy",
  "defense": "fortitude",
  "activation": "action",
  "check": "required",
  "mode": "standard",
  "versions": [
    { "rank": 1, "name": "…", "range": {}, "damage": "1/2/3/5", "description": "…" },
    { "rank": 2, "name": "…", "range": {}, "damage": "2/4/6/9", "description": "…" },
    { "rank": 3, "name": "…", "range": {}, "damage": "4/6/9/14", "description": "…" },
    { "rank": 4, "name": "…", "range": {}, "damage": "5/8/12/18", "description": "…" }
  ]
}
```

Инварианты массива:

- ровно четыре записи;
- ранги уникальны и покрывают `1..4`;
- версия является законченной записью и не складывается с младшими версиями;
- `mode`, `skill`, `defense`, `activation` и `check` общие, пока канон явно не разрешит
  менять их между версиями.

При смене ранга Item не заменяется и не переписывается. Лист, бросок и чат-карта получают
производную проекцию версии нужного ранга. Имя Item остаётся стабильным именем способности;
отображаемое имя берётся из активной версии.

## 6. Стоимость Накала

Текущая стоимость форсированной способности является производной и не сохраняется ни на
Actor, ни на Item:

```text
standard                                     → 0
forced, abilityRank == actorRank             → 2
forced, abilityRank == actorRank − 1         → 1
forced, abilityRank <= actorRank − 2         → 0
```

Поле `mode` остаётся `forced`, даже когда текущая стоимость стала `0`.

## 7. Публичные методы

Стабильный namespace runtime: `game.projectAndromeda.rules`.

```js
getAbilityHeatCost({ mode, abilityRank, actorRank }) // 0 | 1 | 2
gainHeat(actor, amount, { reason })
spendHeat(actor, amount, { source })                 // Promise<boolean>
convertGloryToHeat(actor)                            // Promise<boolean>
resetCombatResources(actor, { phase })               // phase: start | end

deriveStressState(actor)
applyStress(actor, amount)                           // временный пул первым
recoverMainStress(actor)
adjustExhaustion(actor, delta)
resetSceneResources(actor, { phase })                // phase: start | end

resolveSignatureVersion(abilityItem, actorRank)
```

Чистые вычисления должны одновременно экспортироваться из helper-модулей для тестов.
Мутирующие методы нормализуют вход, используют `actor.update(..., { render: false })` и
возвращают итоговое нормализованное состояние либо `boolean` для отклоняемой операции.

`spendHeat` не допускает отрицательного остатка. `convertGloryToHeat` проверяет наличие
Очка Свершения и `gloryConversionUsed`, затем одной логической операцией списывает `1` ОС,
добавляет `2` Накала и ставит флаг.

## 8. Стратегия миграции

Миграция 0.5 является идемпотентной и повторяемой. Одноразовый version gate нужен только
для производительности, а не как гарантия корректности.

```js
game.projectAndromeda.migrations.v05.plan()
game.projectAndromeda.migrations.v05.run({ dryRun: false, force: false })
```

Требования:

1. Запускает только primary active GM.
2. `plan()` ничего не меняет и возвращает полный отчёт.
3. Каждый шаг проверяет фактическое состояние документа, а не только world setting.
4. Повторный запуск не создаёт дубликатов сигнатурок и не переписывает уже нормализованные
   значения.
5. При частичной ошибке setting версии не повышается; повтор продолжает с фактического
   состояния.
6. World schema version записывается только после успешного завершения всех шагов.
7. Неопознанные homebrew-архетипы и сигнатурки сохраняются и попадают в отчёт.
8. Legacy-предметы помечаются до refresh/cleanup компендиума, чтобы последующие задачи не
   удалили их как осиротевшие.

`run()` возвращает количества изменённых Actor/Item, legacy-предметов, нормализованных
сигнатурок, пропусков и ошибок.

## 9. Версия и публичные маршруты

Версия реализации — `0.4.05.0`. Текущая версия `0.4.04.4`; крупный механический реворк
повышает третий компонент и сбрасывает четвёртый. Release line `0.4` не меняется.

Каталог `Книга правил v0.4` не переименовывается. Сохраняются
`RULEBOOK_DIRNAME = "Книга правил v0.4"` и существующие публичные slug `rulebook/...`.
Будущее переименование требует отдельного решения, алиасов, редиректов и проверки внешних
ссылок.

## 10. Отклонённые варианты

- **Переименовать `flux` в `heat`:** семантическое соответствие старых данных не доказано.
- **Использовать `temphealth` или `forceShield`:** старые поля имеют другую семантику и
  смешивают постоянные модификаторы с отдельным посценным пулом.
- **Преобразовать все старые предметы:** разрушительно для UUID и небезопасно для homebrew.
- **Хранить четыре сигнатурки отдельными Item:** создаёт дубликаты, смену UUID и риск
  рассинхронизации.
- **Материализовать текущую стоимость Накала:** значение устаревает при смене ранга.
- **Строго одноразовая миграция:** частичный сбой оставляет мир в невосстановимом смешанном
  состоянии.

## 11. Требует подтверждения владельца

Перед реализацией необходимо явно подтвердить:

1. Не переносить `system.flux.value`, а всегда начинать `system.heat.value` с `0`.
2. Сохранить `weapon` / `armor` / `equipment` как legacy и отключить всю их автоматику
   вместо автоматического преобразования.
3. Использовать идемпотентную миграцию с version gate.
4. Распространить ли ту же legacy-политику на `equipment-consumable`: канон 0.5 не
   предполагает отдельной механической записи для обычных расходников, но этот тип не был
   прямо назван в исходном запросе.
5. Крайний случай `Истощение >= базового максимума`: буквальное чтение канона означает
   немедленное выбытие даже при незаполненных временных ячейках. До подтверждения реализация
   не должна молча выбирать другое поведение.

четырёх рангах: стоимость Накала `2/1/0`, переключение версии сигнатурки, порядок заполнения
двух пулов стресса, складывание/снятие Истощения и повторный запуск миграции.
