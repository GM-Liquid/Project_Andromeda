# Дизайн: редизайн карточки броска под сдвиг ступеней и ступенчатые эффекты

**Дата:** 2026-06-27
**Статус:** утверждён к реализации
**Версия системы:** `0.4.00.1 → 0.4.00.2` (разряд FOUNDRY)

## 1. Контекст и цель

Система проверок изменилась: само выпавшее на кубах число больше не несёт первичной
информации. Игроки и ГМ оперируют **ступенью успеха** и **рангом навыка**, а сдвиги
«ранг против сложности задачи» применяются вручную после броска.

Цель — переделать карточку броска в чате так, чтобы:

- **первично** показывались ступень успеха и ранг навыка;
- **вторично** — урон и эффект для текущей ступени (описание предмета);
- **третично** (свёрнуто) — формула, сумма, выпавшие кубы, модификаторы;
- сдвиг ступени стал **интерактивным**: его можно двигать прямо на карточке, а урон и
  эффекты пересчитываются под итоговую ступень.

Дополнительно: убрать ступень `FailureWithConsequence` (её в новой системе нет) и добавить
модель **ступенчатых эффектов** у предметов, с протаскиванием в JSON-каталоги и Quartz.

## 2. Модель исхода (4 ступени)

Лестница из четырёх ступеней, индексы 0–3, совпадают с порядком значений профиля урона:

| Индекс | Ключ исхода        | RU                  | EN                  |
| ------ | ------------------ | ------------------- | ------------------- |
| 0      | `Failure`          | Провал              | Failure             |
| 1      | `SuccessWithCost`  | Успех с ценой       | Success with a cost |
| 2      | `Success`          | Успех               | Success             |
| 3      | `CriticalSuccess`  | Критический успех   | Critical success    |

- `getSkillCheckOutcomeKey(total)` в [skill-check.mjs](module/helpers/skill-check.mjs) уже
  возвращает ровно эти 4 ключа — менять не нужно.
- **Удалить `FailureWithConsequence`** из `lang/en.json`, `lang/ru.json` и из любых ссылок.
  Эта ступень нигде не генерируется (только висит строкой), удаление чистое. При реализации —
  `grep` по `FailureWithConsequence` во всём репо, чтобы не осталось висячих ссылок.

## 3. Карточка броска (компоновка «циферблат»)

Перерабатывается генератор флейвора в [roll-card.mjs](module/helpers/roll-card.mjs)
(`buildSkillCheckRollFlavor` / `buildSkillCheckRollFlavorFromData`) и CSS-блок
`.myrpg-roll-card*` в [project-andromeda.css](css/project-andromeda.css).

Структура карточки сверху вниз:

1. **Шапка:** название проверки + бейдж **«Ранг N»** (ранг навыка из `skillCheck.rank`).
2. **Дил (первичный блок):**
   - слева вертикальные кнопки **▲ / ▼** (сдвиг на одну ступень за клик);
   - по центру крупная надпись текущей ступени, окрашенная по ступени
     (fail `#8f2f27`, cost `#b07124`, success `#1f7a4d`, crit `#0a6c84`), и подпись:
     - если сдвига нет: `выпало: <ступень>`;
     - если сдвиг есть: `выпало: <выпавшая> · сдвиг ±N`;
   - справа крупное число **урона** для текущей ступени.
3. **Базовое описание** предмета/способности (всегда видно, если задано) — текущее поле `note`.
4. **Панель эффектов** (если у предмета есть `stepEffects`): список строк, каждая —
   `[иконка-статус] <текст эффекта> · <порог>`. Эффект **активен**, если
   `текущая ступень >= порог` (накопительно): активные подсвечены, неактивные приглушены/«заперты».
5. **`<details>` «Детали броска»** (третично, свёрнуто): формула `2d8 + @mod`, сумма, выпавшие
   кубы, модификаторы, исходный профиль урона.

Урон к броску **не прибавляется**: показывается одно из значений `system.skillBonus`
(профиль `0/1/2/4`) по индексу текущей ступени через `getDamageForOutcome`. Если профиль
`0/0/0/0` — блок урона скрыт (как сейчас, через `hasDamageProfileValue`).

## 4. Сдвиг: интеракция, хранение, права

### 4.1 Хранение

- В `flags.project-andromeda.skillCheck` добавляется поле **`shift`** — целое смещение
  от выпавшей ступени (по умолчанию `0`).
- При каждом рендере карточка вычисляет:
  - `rolledIndex = индекс(getSkillCheckOutcomeKey(total))`;
  - `currentIndex = clamp(rolledIndex + shift, 0, 3)`;
  - `outcomeKey` для урона/эффектов берётся из `currentIndex`.
- Сдвиг хранится как **дельта**, а не как абсолютная ступень: это сохраняет корректность,
  когда «потратить героизм» меняет `total` (выпавшая ступень пересчитывается, дельта остаётся).

### 4.2 Интеракция и перерисовка

- Клик по ▲/▼ меняет `shift` (▲ `+1`, ▼ `−1`), вызывает `message.update` флага и
  перестраивает флейвор тем же путём, что «потратить героизм»
  (см. `buildSkillCheckRollFlavorFromData` и перестройку флейвора в
  [project-andromeda.mjs](module/project-andromeda.mjs) ~строки 346–352).
- Обработчик кликов вешается в хуке рендера сообщения чата (там же, где уже навешан
  контекст-экшен героизма).

### 4.3 Права (упрощённая модель «автор + ГМ»)

- Двигать ступень может **автор броска** (владелец сообщения) и **ГМ**.
- Прочим игрокам кнопки ▲/▼ либо не показываются, либо неактивны (disabled).
- Socket-релей «кто угодно двигает чужую карточку» **не делаем** — это сознательное
  упрощение, чтобы не вводить серверную пересылку обновлений.

## 5. Ступенчатые эффекты: модель данных и ввод

### 5.1 Поле данных

Новое поле на предмете — **`system.stepEffects`**: массив объектов

```
{ text: string, minOutcome: "SuccessWithCost" | "Success" | "CriticalSuccess" }
```

- `text` — произвольная строка эффекта («Оглушение на 1 раунд»);
- `minOutcome` — минимальная ступень срабатывания; порог `Failure` не предлагается
  (провал ничего не включает). Допустимые пороги — три «успешных» ступени.
- Эффект активен на карточке, когда `currentIndex >= индекс(minOutcome)`.
- Пустой массив → панель эффектов скрыта.

### 5.2 Типы предметов

`stepEffects` добавляется тем типам, где уже есть профиль урона / `requiresRoll`:
**оружие (`weapon`), способности/геномы (`ability`/genome), снаряжение (`equipment`) с
`requiresRoll`**. Конкретные схемы — по фактическому наличию `system.skillBonus`.

### 5.3 Авторинг на листе предмета

- Рядом с полем профиля урона (`system.skillBonus`) — **повторяемый список строк**:
  текстовое поле + выпадашка порога (3 ступени) + кнопки `+` / удалить.
- Затрагиваемые шаблоны: как минимум [weapon-sheet.hbs](templates/item/weapon-sheet.hbs),
  а также шаблоны способностей и снаряжения-с-броском (по факту наличия поля урона;
  [implant-sheet.hbs](templates/item/implant-sheet.hbs),
  [cartridge-sheet.hbs](templates/item/cartridge-sheet.hbs),
  [generic-sheet.hbs](templates/item/generic-sheet.hbs) — проверить при реализации).
- Логика добавления/удаления строк — в [item-sheet.mjs](module/sheets/item-sheet.mjs),
  по образцу уже существующих повторяемых полей. Сабмит с `render: false` там, где уместно.

## 6. Протаскивание в JSON-каталоги и Quartz

Изменение модели данных должно дойти до каталогов снаряжения и публичной книги правил.

- **`template.json`** — добавить `stepEffects: []` в схемы соответствующих типов предметов.
- **`data/gear/catalog/*.json`** (публичный fallback-mirror) — расширить схему: опциональный
  `stepEffects` у записей. Обратная совместимость: поля нет → пустой массив.
- **[gear-catalog.mjs](module/helpers/gear-catalog.mjs)** — трансформ JSON→Item переносит
  `stepEffects` в `system`; тогда `npm run build:pack` кладёт эффекты в компендиум `gear-library`
  автоматически, без отдельной правки `build-pack`.
- **Quartz, глава 04** — [sync-book.mjs](.quartz-site/scripts/sync-book.mjs): в генерируемых
  таблицах снаряжения отобразить ступенчатые эффекты (отдельной строкой/колонкой под предметом).
- **Канон в приватном репо:** канонический источник — `Docs_Project_Andromeda/data/gear/catalog/`.
  В этой сессии правится только **публичная схема/fallback** в этом репозитории; синхронное
  обновление приватного каталога остаётся за мейнтейнером (напомнить отдельно).

## 7. Локализация

Все новые строки — одновременно в `lang/en.json` и `lang/ru.json` с идентичными ключами:

- заголовок панели «Эффекты» / «Effects»;
- метки порогов (переиспользовать `MY_RPG.SkillCheck.Outcomes.*`);
- подписи дила: «выпало» / «rolled», «сдвиг» / «shift»;
- метки авторинга `stepEffects` на листе предмета (заголовок списка, плейсхолдер текста,
  лейбл выпадашки порога, кнопка добавления).

Удалить ключ `MY_RPG.SkillCheck.Outcomes.FailureWithConsequence` из обоих файлов.

## 9. Версия

Правка трогает shipped-файлы (`module/`, `templates/`, `lang/`, `template.json`, `css/`),
поэтому `system.json` бампается. По §4 третий разряд (RULES) меняется только по прямому
указанию мейнтейнера — указания не было, поэтому берём **FOUNDRY**: `0.4.00.1 → 0.4.00.2`.

## 10. Вне рамок (YAGNI)

- Никакого socket-релея для «кто угодно двигает чужую карточку».
- Никакой автоматизации применения эффектов к токенам/состояниям (эффект — просто текст).
- Никаких тег-типов/иконок у эффектов (отклонённый вариант ввода).
- Никакого числового под-профиля у эффектов (отклонённый вариант ввода).
- Никакой миграции старых сообщений чата: у старых сообщений `shift` отсутствует → трактуется
  как `0`, карточка перестроится при следующем рендере.
- Обновление приватного канонического каталога `Docs_Project_Andromeda` — вне этой сессии.
